# FE Audit Report — 2026-05-13

Reference spec: `docs/superpowers/specs/2026-05-13-fe-audit-design.md`

## Executive Summary

- **Biggest issue — no FE gate in CI.** The only CI workflow builds and deploys the Rust binary; no step runs `pnpm install`, `pnpm build`, `pnpm astro check`, or `pnpm format:check`. A type error already exists in `web/pages/project.astro` that `astro check` catches and CI never surfaces. Vercel's build check runs after merge, not before — formatting and type regressions are invisible until they land on `master`.
- **Biggest surprise — the Rust backend fetch is eliminable.** `OnePiece.tsx` fetches `GET /api/op/today` at runtime to get today's character, but the Rust backend seeds a deterministic PRNG from the current UTC date and picks a character index from the same `characters.json` that is already bundled into the JS chunk at build time. The computation is fully reproducible client-side. Removing the fetch eliminates the Lightsail dependency for the `/op/` route and resolves the dead `loading`/`error` state that currently shows nothing to the user when the backend is down.
- **Biggest performance liability — 4.4 MB of fonts unconditionally preloaded on every page.** Pages with no CJK text (`/contact/`, `/project/`, `/cv/` in English) still instruct the browser to download two SourceHanSerifSC subsets totalling 3.3 MB. This is the dominant first-visit cost on a cold cache for every route on the site.
- **What's healthy and should not be touched.** The overall codebase is lean and well-structured: 26 files, one real `any`, zero `@ts-ignore`, strong separation between the `op/` sub-package (types / utils / translations / component), and a Tailwind setup that is correct under v3. The content collections schema is sound for five of six collections. The build is fast (2.67 s) and every page outside `/op/` ships zero JavaScript.

### Notes

#### `@astrojs/tailwind` → `@tailwindcss/vite` (the Tailwind v4 migration story)

`@astrojs/tailwind` has not received Tailwind v4 support and is effectively superseded. Astro's official documentation now directs users to `@tailwindcss/vite` as the replacement: the npm `deprecated` field on the package is not set, and the README carries no formal deprecation notice, but the integration is functionally at end-of-life for v4 workflows. Astro's own docs (`guides/styling.mdx`) describe the exact migration path:

1. Run `pnpm astro add tailwind` (Astro 5.2.0+) — this installs `@tailwindcss/vite` and `tailwindcss@4`, wires the Vite plugin into `astro.config.mjs`, and adds `@import "tailwindcss"` to a global CSS file.
2. `pnpm remove @astrojs/tailwind` and remove the `tailwind()` call from `astro.config.mjs`.
3. Follow the [official Tailwind v4 upgrade guide](https://tailwindcss.com/docs/upgrade-guide) to convert `tailwind.config.mjs` config (theme extensions, custom fonts, colours, typography plugin, keyframes) to CSS-first `@theme` blocks.
4. Replace `require("@tailwindcss/typography")` in the config with `@plugin "@tailwindcss/typography"` in CSS.
5. Bump `prettier-plugin-tailwindcss` to `^0.8.0` for v4 class-sorting support.

The project's `tailwind.config.mjs` is moderately complex (custom fonts, colours, five `typography` variants, keyframes) — the config-to-CSS conversion is mechanical but not trivial.

#### Unused / phantom deps

- **`@vercel/analytics` should be removed.** It is declared in `dependencies` but is never imported anywhere in the codebase. Vercel Analytics is already active through the `webAnalytics: { enabled: true }` option on the `@astrojs/vercel` adapter, which injects the tracking script at build time. The standalone package is a dead listing that inflates `node_modules` for no reason.

#### `@astrojs/mdx` loaded without any `.mdx` content

`@astrojs/mdx` is wired in `astro.config.mjs` and the integration is legitimate (it also extends Markdown rendering), but there are currently **zero `.mdx` files** in `web/`. All 23 content posts are `.md` files.

**Open question:** Are any `.mdx`-authored content surfaces planned? If no, the integration should be removed. If yes, no action needed — bump to 5.x as part of the Astro 6 migration.

**Finding — oversized OnePiece bundle (medium):** `/op/` ships 142 KB of JS (gzip: `client.BJjWi__I.js` 56.2 KB + `OnePiece.CT4baDkq.js` 85.8 KB) for a route that is entirely opt-in — only users who navigate to `/op/` are affected, unlike the universal CJK font preload. This is the entire React DOM runtime bundled with `@headlessui/react`, `ts-pattern`, and `tiny-pinyin` — all loaded synchronously on `client:load`. No other page ships any JS at all. The `client.BJjWi__I.js` chunk (56.2 KB gzip) is the complete React DOM + scheduler; it is only ever needed on `/op/`, which is expected for a React island but means the `/op/` route is the sole source of all runtime JS weight. Recommendation: Profile the OnePiece bundle with `vite-bundle-visualizer` (or `rollup-plugin-visualizer` via Astro's vite config) to attribute size to specific deps. Likely culprits: `@headlessui/react` (Combobox/Listbox/Dialog), `tiny-pinyin`, and the React DOM runtime. Concrete next steps: replace HeadlessUI Combobox/Listbox with native `<select>` where possible; lazy-load Dialog panels via `React.lazy` + `Suspense`; gate `tiny-pinyin` behind the search-input handler instead of importing at module top-level.

**Finding — CSS shipped to all pages regardless of use (low):** The single 7.3 KB (gzip) CSS bundle is emitted to every route including `/contact/` (plain text page). This is a known Tailwind v3 characteristic — the full purged stylesheet is one file. Tailwind v4 with its per-file CSS approach would split this more naturally.

### Font payload

Five font files are served from `/fonts/`, all with `rel="preload"` injected on every page via `web/components/Base.astro` lines 33–55.

| File | Size on disk | Role |
|------|-------------|------|
| `jbmono-regular.woff2` | 90 KB | JetBrains Mono regular — UI monospace font |
| `jbmono-bold.woff2` | 92 KB | JetBrains Mono bold |
| `NotoSerifSC-ExtraBold-subset.woff2` | 926 KB | CJK display font — content-subsetted |
| `SourceHanSerifSC-Regular-subset.woff2` | 1.6 MB | CJK body font — content-subsetted |
| `SourceHanSerifSC-Medium-subset.woff2` | 1.7 MB | CJK body font medium — content-subsetted |
| **Total** | **4.4 MB** | |

**Subsetting status:** The three CJK fonts are subsetted by `src/bin/sync_fontbook.rs`, which collects all characters from `docs/blog/*.md` and `docs/salt/*.md` source files, adds a fixed ASCII+punctuation baseline, then calls `pyftsubset` via `fonttools`. The script writes a `.char_count` marker to `public/fonts/` after each run. That marker file is **missing** from the current checkout, which means either the subsetting step has not been run since the last clean, or the file was not committed. The woff2 files themselves are present and their sizes are consistent with content-subsetted CJK fonts (full unsubsetted SourceHanSerifSC weighs ~25 MB), so subsetting appears to have run at some prior point. However, because the sync_fontbook script reads from `docs/blog/` (source) rather than `web/content/posts/` (synced copy), any new posts added directly to `web/content/` without running `sync_content` first would not be covered by the next subsetting run.

**Finding — 4.4 MB font preload on every page including non-CJK pages (high):** All five fonts are unconditionally preloaded (`rel="preload"`) in `web/components/Base.astro:33–55` and declared via `@font-face` in `web/styles/global.css:5–46`. Pages like `/contact/` and `/project/` that contain no CJK text still instruct the browser to download 4.4 MB of fonts. Two of those files (SourceHanSerifSC-Regular and SourceHanSerifSC-Medium, totalling 3.3 MB) are the dominant cost. Even with Vercel CDN and HTTP/2, this is the largest single performance liability on the site for first-time visitors whose cache is cold. Recommendation: conditionally preload the CJK fonts only on pages that actually use CJK content (blog posts, book notes, CV) — or switch to `font-display: swap` + no preload so they load lazily on pages that render them via CSS.

**Finding — JetBrains Mono not content-subsetted (low):** `jbmono-regular.woff2` (90 KB) and `jbmono-bold.woff2` (92 KB) are not content-subsetted. JBMono covers a large glyph range. For a site primarily using it as a UI/code font, a subset to Latin + code punctuation + common Unicode symbols could cut these to ~30–40 KB each. The `sync_fontbook.rs` script only subsets the three CJK fonts; JBMono is shipped as-is. Recommendation: add JBMono to the fontbook subsetting workflow or use a CDN-hosted, variable-weight version.

**Finding — missing `.char_count` sentinel (low):** `public/fonts/.char_count` is absent. This file is the only indicator that `sync_fontbook` has been run since the last clean. Without it, there is no runtime check to verify the subsets are fresh. Recommendation: commit the `.char_count` file or add a build-time check that aborts the build if it is missing (to prevent shipping stale subsets after a clean).

### Image strategy

No `astro:assets` (`<Image />` or `<Picture />`) components are used anywhere in the codebase — confirmed by grepping all `web/` `.astro` files for `from "astro:assets"`, `<Image`, and `<Picture`. All image references are either:

1. **Raw `<img>` tags in Markdown content** (no processing): 10 occurrences across 3 blog posts:
   - `web/content/posts/Miller-Rabin.md:208` — `<img src="/images/52536/dual_e.png">`
   - `web/content/posts/Fetching Github Stats using GraphQL in React.md:15,21,126,129,132` — 5 `<img>` tags referencing `/images/github_graphql/`
   - `web/content/posts/Build a Web App 1.md:123,179,220,262` and `Build a Web App 2.md:86` — 5 `<img>` tags referencing `/images/73316/`

2. **Raw `<img>` tags in the OnePiece React component** (`web/components/op/OnePiece.tsx:136,430,491,591`) — loading character images from an external URL via `char.image` prop (remote URLs from One Piece wiki, fetched at runtime).

3. **Hardcoded `/public/` paths in `<link>` and `<meta>` tags** (apple-touch-icon variants, og:image set to `/favicon.png`) in `web/components/Base.astro:21–29,73–75`.

None of these paths go through Astro's image optimization pipeline, which means: no automatic format conversion to WebP/AVIF, no `srcset` generation, no dimension enforcement, no lazy loading via the framework (the OnePiece `<img>` tags do use `loading="lazy"` manually).

**Finding — zero use of astro:assets (medium):** The site ships `.png` and `.jpg` files with no format optimization or responsive variants. The blog post images in `public/images/` (totalling ~750 KB across 3 directories) are served as-is. `<Image />` from `astro:assets` would auto-convert to WebP and add width/height to prevent layout shift. Recommendation: migrate the 3 blog-post image directories to use `astro:assets` imports or at minimum add explicit `width` and `height` attributes to prevent CLS. For the OnePiece component, external remote images are harder to optimize but could benefit from Vercel's image optimization proxy.

**Finding — orphaned `bamboo.jpg` in public/ (low):** `public/bamboo.jpg` (30 KB) exists in `public/` but is not referenced by any source file in `web/`. Recommendation: remove it.

**Finding — MathJax loaded globally on all pages (medium):** `web/components/Base.astro:85–89` unconditionally loads MathJax from CDN (`https://cdn.jsdelivr.net/npm/mathjax@3/es5/tex-mml-chtml.js`) on every page via `async` script tag. Only 1 of 23 blog posts (`Miller-Rabin.md`) uses math notation (`$$...$$`, `\(...\)` LaTeX syntax). Every other page (home, /book/, /cv/, /contact/, /op/) loads a CDN resource it never uses. The CDN script is `async` so it does not block rendering, but it still adds a DNS lookup and network request on every page load, and depending on browser preconnect behavior can affect LCP. Recommendation: move the MathJax script tag into the blog post layout template and gate it on a frontmatter flag (e.g., `math: true` in the post's YAML), rather than loading it site-wide.

### OnePiece.tsx deep dive

`OnePiece.tsx` is 747 lines. The work it is doing:

1. **State management** — 5 `useState` calls: `language` (en/cn toggle), `query` (search input), `guessHistory` (ordered list of guessed characters), `todaysCharacter` (fetched from API), `loading`/`error` (fetch lifecycle). `loading` and `error` are set but never read in the render tree (see type-tightness section).
2. **Local-storage persistence** — Two `useEffect` hooks serialize/deserialize game state (date + guess history names) and language preference to `localStorage`. The guard logic is correct: on a new day the stale state is cleared.
3. **External DOM coupling** — One `useEffect` listens for clicks on a `#language-toggle` button that lives outside the React root in the Astro `Callout` slot (`pages/op/index.astro`). The component uses `document.getElementById` to wire the event listener. This is a direct DOM escape hatch that bypasses React's event system — it works but creates a coupling that is invisible to React's render cycle.
4. **Runtime fetch** — One `useEffect` calls `fetch(`${API_BASE_URL}/api/op/today`)` to get today's character from the Rust backend. The backend (`src/onepiece.rs`) seeds a `StdRng` from the current UTC date (`year * 10000 + month * 100 + day`) and picks a deterministic character index (verified in src/onepiece.rs:67–78). This value is computable client-side from the same algorithm using the same `characters.json` that is already bundled into the JS chunk at build time (`import charactersData from "@/content/onepiece/characters.json"` — line 7). The runtime fetch is therefore **eliminable**: the same daily-seeded random pick could run entirely in the browser using the same `characters.json` import, removing the need for the Rust backend for this feature.
5. **Multi-language handling** — Language (`en` / `cn`) is a top-level state value passed down to `useTranslation(language)` which returns an overloaded `t()` function, and to `getHakiDisplay(char, language)`. All language-specific field lookups are centralized in `translations.ts` via the `characterFieldGetters` map. The isolation is clean: no language switch logic leaks into `utils.ts` (pure comparison functions). The one exception is `getHakiDisplay` in `translations.ts:95–107`, which receives `language` as an argument — this is acceptable, not a structural leak.
6. **Duplicated rendering between desktop table and `MobileGuessCard`** — The desktop `<table>` body (lines 577–729) and `MobileGuessCard` component (lines 111–279) render the same 9 fields per character with the same comparison logic and the same color-coding (9 colored/compared fields; the desktop table also shows non-compared columns for index, image, and name). The comparison calls (`compareAffiliation`, `compareStrings`, etc.) are repeated verbatim in both branches. This is pure duplication: a shared `CharacterCellData` function that returns `{ value, matchClass, directionArrow }` tuples for each field would allow both renderers to share the business logic.

**Verdict — refactor candidate (not replace):** The architecture of the op sub-package is sound: types, utils, and translations are cleanly separated. The runtime fetch to `/api/op/today` is the primary structural problem — it requires a running Rust backend for a value that is deterministically derivable client-side from already-bundled data. Eliminating the fetch would also resolve the `loading`/`error` dead-state issue. To eliminate the fetch: port `get_todays_character` to TypeScript — seed a deterministic PRNG with `year * 10000 + month * 100 + day` (UTC), generate one index in `[0, characters.length)`, and index into the already-imported `characters` array — then delete the `useEffect` fetch and the `loading`/`error` state entirely. The desktop-vs-mobile duplication is a secondary refactor opportunity. The external DOM coupling for the language toggle (reaching outside the React root to wire `#language-toggle`) is a design smell that would be cleanly resolved by making `language` a prop passed down from the Astro page level or by moving the toggle button inside the React component.

### Schema audit

Content collections defined in `web/content.config.ts`.

#### `posts` collection

Schema fields: `title`, `displaytitle`, `abbrlink`, `date`, `disabled`.

`z.coerce` usage:
- `title: z.coerce.string()` — **verdict: fix upstream.** The sample post frontmatter has `title: 去开会成为一个人新生活的标志` (plain string). All 23 posts have string titles in YAML. `z.coerce.string()` would convert numbers/booleans to strings if present, but no such case exists. Use `z.string()` directly.
- `displaytitle: z.coerce.string().optional()` — **verdict: fix upstream.** Same reasoning as `title`; this field is freeform text and is always a string literal when present.
- `abbrlink: z.coerce.number()` — **verdict: legitimate.** YAML frontmatter emits `abbrlink` as a bare integer (e.g., `abbrlink: 25397`). Astro's YAML parser may deliver this as a string in certain contexts; `z.coerce.number()` is a defensive guard. Keeping it is acceptable, but if the YAML parser consistently delivers a number, `z.number()` would be stricter.
- `date: z.coerce.date()` — **verdict: legitimate.** YAML datetime strings (e.g., `date: 2022-07-20 03:08:11`) are not native JS `Date` objects; coercion is required.

`disabled: z.boolean().optional().default(false)` — field is absent from all checked post frontmatter, which means `.optional().default(false)` is correct: unset posts are not disabled by default. No issue.

Cross-reference against sample (`去开会成为一个人新生活的标志.md`): frontmatter has `title`, `abbrlink`, `date` — no `displaytitle`, no `disabled`. Schema handles both gracefully.

#### `booknotes` collection

Schema fields: `title`, `author`, `id`, `notenum`, `rating`, `tags`, `content`.

`z.coerce` usage:
- `title: z.coerce.string()` — **verdict: fix upstream.** Booknote file (`国境以南_太阳以西.json`) has `"author": "村上春树"` — title is the filename slug used as ID. In the JSON the title is not a top-level field (it is absent from the sample JSON), meaning `title` is inferred from the filename by Astro's glob loader. Whether this arrives as a string or something else depends on the loader. However, JSON string values are always strings — `z.string()` should suffice.
- `rating: z.coerce.number()` — **verdict: legitimate.** JSON numbers are parsed as numbers by the JSON loader, so `z.number()` would also work, but `z.coerce.number()` is harmless for numbers.

`id: z.number()`, `notenum: z.number()` — not coerced, correct (JSON integers come through as numbers).

`author: z.string()` — not coerced, correct.

Cross-reference against sample JSON: has `author`, `content` array. `title`, `id`, `notenum`, `rating`, `tags` are not present as top-level keys in the JSON file — these appear to be populated by metadata injected at sync time (via `src/bin/gen_booknote.rs` or similar). A full booknote JSON file would need to be inspected to confirm; the sample used here is truncated.

**Finding — booknote schema missing `id`, `notenum`, `rating`, `tags` in sample JSON (low):** The checked booknote JSON (`国境以南_太阳以西.json`) does not include `id`, `notenum`, `rating`, or `tags` as top-level keys — only `author` and `content` were visible in the head. If the full file omits these fields, the schema would fail validation at build time. This warrants inspecting a complete booknote JSON to verify the shape matches the declared schema.

#### `experiences` collection

Schema fields: `company`, `role`, `date`, `location`, `description?`, `points?`, `is_internship?`, `link?`, plus a `.transform()` that parses `date` string into `startDate: Date`.

No `z.coerce` usage. `date: z.string()` accepts the string `"Jul 2025 - Present"` and the transform converts it to a `Date` for sorting.

The transform's `parseDate` helper uses `str.trim().split(" ")` to extract month and year from strings like `"Jul 2025"` or `"Present"`. For `"Present"` (no year part), `parts.length > 1` is false, so it falls back to `fallbackYear` derived from `endStr`. If `endStr` is `"Present"`, no year is found in the regex `\d{4}`, so `fallbackYear` stays as `new Date().getFullYear()`. This is correct behavior: current-role entries get `startDate` in the current year. No issue.

`link: z.string().url().optional()` — heygen.yaml has `link: https://app.heygen.com/`, which passes URL validation. `description` and `points` are `.optional()`, which is appropriate since some entries have neither.

**Finding — `is_internship` default may mask missing field (low):** `is_internship: z.boolean().optional().default(false)` — this silently defaults to `false` for all entries that omit the field, including future ones. This is the intended behavior for a binary flag, but if an author forgets to set it, no schema error will surface. Low risk, acceptable pattern.

#### `skills` collection

Schema fields: `name`, `id`, `items`.

No `z.coerce` usage. All fields are required. Sample (`language.yaml`) has `name: Languages`, `id: 1`, `items: [...]` — matches exactly. No issues.

#### `projects` collection

Schema fields: `name`, `id`, `language`, `description`, `link?`.

No `z.coerce` usage. `link: z.string().optional()` — no URL validation unlike `experiences.link`. Sample (`bamboo.yaml`) has `link: https://github.com/k27dong/bamboo`.

**Finding — `projects.link` not URL-validated (low):** Unlike `experiences.link` which uses `z.string().url()`, the `projects` collection's `link` field uses plain `z.string()`. A non-URL value would pass schema validation. Recommendation: change to `z.string().url().optional()` to match the experiences schema pattern.

#### `onepiece` collection

Schema is a `z.array(...)` wrapping per-character objects. 279 characters in `characters.json`, all data verified present.

Required fields in Zod schema: `name`, `japanese_name`, `image`, `debut_chapter`, `debut_arc`, `affiliations`, `origin`, `bounty`, `status`, `haki`, `cn` (with `cn.name`, `cn.affiliations`, `cn.origin`, `cn.debut_arc`, `cn.haki` required).

Optional fields: `occupations`, `residence`, `age`, `birthday`, `height`, `devil_fruit_name`, `devil_fruit_type`, plus `cn.devil_fruit_name`, `cn.devil_fruit_type`.

No `z.coerce` usage in onepiece schema.

**Schema vs TypeScript type mismatch (medium):** The Zod content collection schema marks `japanese_name`, `debut_chapter`, `debut_arc`, `affiliations`, `origin`, `bounty`, `status`, `haki`, and `cn` as **required**. The TypeScript `Character` interface in `web/components/op/types.ts` marks `japanese_name`, `debut_chapter`, `debut_arc`, `affiliations`, `origin`, `bounty`, `status`, `haki` as **optional** (`?:`). The interface is what `OnePiece.tsx` uses at runtime — the component defensively guards these fields with `|| ""` and `|| []` patterns. Since the game reads `characters.json` via a direct import (not through the content collection loader), the Zod schema's validation does not apply at runtime; the TypeScript interface is the operative contract. The interface is deliberately looser than the data, which is correct for runtime safety but means the two type definitions are maintained separately and can drift. Recommendation: derive the `Character` interface from the Zod schema using `z.infer<typeof characterSchema>` so there is a single source of truth.

**Finding — `bounty` is required in schema but semantically optional (low):** Zod schema has `bounty: z.number()` (required, non-optional). 127 of 279 characters have `bounty: 0`, representing characters with no known bounty. Using `0` as the sentinel for "no bounty" prevents distinguishing between a character with a confirmed zero bounty and one whose bounty is unknown. The TypeScript interface correctly uses `bounty?: number` (optional), suggesting the original intent was to use `undefined` for unknown bounties. As long as the data file is the source of truth and bounty 0 is meaningful, this is acceptable — but a future schema that marks bounty `optional` and migrates 0-bounty entries to `undefined` would be more semantically precise.

**Finding — `start` is a duplicate of `dev` (low):** Both `start` and `dev` run `astro dev`. `start` adds no value beyond `dev` and may confuse contributors who expect `start` to run a production server. Recommendation: remove `start` or repurpose it to `astro preview` (which mirrors the production output).

**Finding — `@astrojs/check` not in devDependencies (medium):** `astro check` internally delegates to `@astrojs/check`. This package is not listed anywhere in `package.json` — neither `dependencies` nor `devDependencies`. When Axis 3 ran `pnpm astro check`, the tool auto-installed `@astrojs/check` on first use. In a frozen-lockfile CI environment (`pnpm install --frozen-lockfile`), this auto-install would fail. Recommendation: add `"@astrojs/check": "^0.9"` (or latest) to `devDependencies` and pin it in the lockfile.

### Missing tooling

#### No `astro check` in CI (high)

The only CI workflow is `.github/workflows/deploy.yml`. It:
1. Checks out the repo
2. Runs `cargo build --release` to compile the Rust backend binary
3. Uses `appleboy/scp-action` to upload the binary to AWS Lightsail via SCP

There is **no FE-related step**: no `pnpm install`, no `pnpm build`, no `pnpm astro check`, no `pnpm format:check`. The Vercel deployment is triggered separately by Vercel's GitHub integration (not by this workflow). This means:

- TypeScript type errors (like the `project.astro` sort callback error) pass CI without any signal
- Formatting regressions are not caught
- A broken Astro build (e.g., a bad import after a refactor) only surfaces when Vercel's build runs, which happens after merge — not before

The Vercel deployment is gated only on Vercel's own build success check on the PR. If `astro check` is never run by CI, the type error found in Axis 3 could persist for months unnoticed.

Recommendation: add a FE job to `deploy.yml` (or a separate `fe-check.yml` workflow) that runs on `push` / `pull_request` to `master`:

```yaml
fe-check:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v6
    - uses: pnpm/action-setup@v3
      with: { version: 9 }
    - uses: actions/setup-node@v4
      with: { node-version: 22, cache: "pnpm" }
    - run: pnpm install --frozen-lockfile
    - run: pnpm astro check
    - run: pnpm format:check
```

This makes type errors and formatting regressions blocking on every PR, not invisible until Vercel builds.

#### Deprecated integration pattern (`@astrojs/tailwind`) (medium)

See `astro.config.mjs` and Axis 1 findings. The `tailwind()` integration from `@astrojs/tailwind` is functional under v3 but is the wrong integration for any future Tailwind v4 upgrade. No action needed before the v4 migration sprint, but the integration should be removed and replaced with `@tailwindcss/vite` as part of that sprint.

#### TypeScript strictness gap (medium)

`tsconfig.json` extends `astro/tsconfigs/base`, which does not enable the `strict` umbrella flag. `noImplicitAny` and `noUncheckedIndexedAccess` are both absent from the effective config. The codebase is small enough that upgrading to `astro/tsconfigs/strict` is tractable. See `tsconfig.json` subsection above for the full effective flags table.

## Axis 5: Content Workflow

### Sync script overview (`src/bin/sync_content.rs`)

The sync script has three functions, each of which **wipes the target directory first** and then repopulates it from scratch.

**`sync_blogposts()`** — Deletes `web/content/posts/`, recreates it, then glob-copies every `docs/blog/*.md` file into the new directory verbatim, preserving only the filename. No transformation is applied: content, frontmatter, and encoding are copied as-is. The `docs/blog/archived/` subdirectory is not matched by the `*.md` glob so its files are silently excluded from the sync.

**`sync_booknotes()`** — Deletes `web/content/booknotes/`, recreates it, then calls `booknote::load_booknote()` (from `sitecore`) to parse all Markdown files under `docs/salt/`. For each parsed booknote, it writes a new JSON file to `web/content/booknotes/<title_with_spaces_replaced_by_underscores>.json`. The JSON shape is `{ title, author, id, notenum, rating, tags, content }` — a transformation from Markdown + YAML frontmatter into a structured JSON document. This is a one-way transform: the JSON is derived from the Markdown, not authored directly.

**`sync_onepiece()`** — Deletes `web/content/onepiece/`, recreates it, then copies `data/op_sanitized.json` to `web/content/onepiece/characters.json`. This is a separate workflow: the source file is not under `docs/` but under `data/`, which is presumably produced by an external scrape/sanitize step. If `data/op_sanitized.json` is absent, the function emits a warning and leaves the directory empty; there is no error exit.

### Drift measurement

#### Blogposts (`docs/blog/` vs `web/content/posts/`)

| Metric | Count |
|--------|-------|
| Files only in `docs/blog/` | 1 directory (`archived/`, 12 files — excluded by the `*.md` glob so this is expected) |
| Files only in `web/content/posts/` | 0 |
| Files present in both, reported as differing | **23 of 23** |

All 23 blog posts are reported as differing. The drift is **entirely due to line endings**: `docs/blog/*.md` files use Unix LF (`0x0A`), while every file in `web/content/posts/` uses Windows CRLF (`0x0D 0x0A`). Stripping carriage returns (`tr -d '\r'`) before diffing produces zero output for all checked files — content is byte-for-byte identical. The CRLF was introduced when `web/content/posts/` was last committed on a Windows machine (or one with `git config core.autocrlf=true`), baking CRLF into the committed objects. `fs::copy` is not the cause — it preserves source bytes faithfully, and the source (`docs/blog/`) has always been LF. Subsequent checkouts on any platform perpetuate the divergence because the CRLF is in the committed data, not injected at checkout time.

In practice this drift is **harmless for the Astro build**: Astro's Markdown parser handles CRLF transparently. However, it means `git diff` after a sync run will always show 23 changed files (line ending churn), making it impossible to use diff-cleanliness as a sync guard without a `.gitattributes` normalization rule.

**Characterization of drift for checked files:**
- `你好，博客.md`: LF in source, CRLF in posts — content identical after CR-strip.
- `Build a Web App 1.md`: LF in source, CRLF in posts — content identical after CR-strip.
- `Miller-Rabin.md`: LF in source, CRLF in posts — content identical after CR-strip.

#### Booknotes (`docs/salt/` vs `web/content/booknotes/`)

| Metric | Count |
|--------|-------|
| `.md` files in `docs/salt/` | 106 |
| `.json` files in `web/content/booknotes/` | 106 |

Counts match exactly. Title correspondence verified: `docs/salt/国境以南 太阳以西.md` (frontmatter `title: 国境以南 太阳以西`, `author: 村上春树`) corresponds to `web/content/booknotes/国境以南_太阳以西.json` (contains `"author": "村上春树"` and a `"content"` array). The space-to-underscore renaming in the JSON filename is consistent with the sync script's `title.replace(" ", "_")` logic. No missing or extra files.

#### Onepiece (`data/op_sanitized.json` → `web/content/onepiece/characters.json`)

`web/content/onepiece/characters.json` is present (last modified 2026-05-13 14:12:46 local time, matching today's audit session). The source `data/op_sanitized.json` does **not exist** in the working tree — the `data/` directory is entirely absent from the repository. This means the onepiece sync cannot be reproduced from the current checkout: `sync_onepiece()` would emit the warning `"Warning: data/op_sanitized.json not found."` and leave `web/content/onepiece/` empty. The characters data is committed directly to `web/content/` and is not regenerable without the external scrape artifact. This is a distinct workflow from the `docs/` → `web/content/` author flow; it is a one-time or ad-hoc scrape result that is committed as-is. Regeneration path: `cargo run --bin sync_onepiece` rebuilds `data/op_sanitized.json` from its raw input (via layers 1–14, where layer 1 scrapes the One Piece Wiki into `data/op_original.json`), but that raw input is also absent from the repo — flag as open question.

### Drift summary

| Content type | Source | Target | Drift kind | Severity |
|---|---|---|---|---|
| Blogposts | `docs/blog/` (23 .md, LF) | `web/content/posts/` (23 .md, CRLF) | Line-ending mismatch — content identical | low |
| Booknotes | `docs/salt/` (106 .md) | `web/content/booknotes/` (106 .json) | None — counts match, titles correspond | none |
| Onepiece | `data/op_sanitized.json` (absent) | `web/content/onepiece/characters.json` (present) | Source file missing from repo; not reproducible from checkout | medium |

### Options for resolving drift and formalizing the workflow

#### Option A: Gitignore `web/content/`, regenerate at every build

Astro's build step would call `cargo run --bin sync_content` first (via a `prebuild` script or Vercel `buildCommand` prefix), then run `astro build`. `web/content/` is removed from git and treated as generated output.

**Pros:**
- `web/content/` is never stale — always a fresh copy of `docs/`.
- Removes 23 CRLF/LF noise files from the git history.
- Single source of truth: authors write only in `docs/`.

**Cons:**
- Vercel's build environment does not have Rust installed by default. A `rust-toolchain.toml` and a custom install step (e.g., `curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh`) would be needed, which significantly complicates the build and slows cold deploys.
- `sync_onepiece()` reads `data/op_sanitized.json`, which is absent from the repo. The onepiece content would be wiped on every build unless the source artifact is committed or fetched separately.
- Increases Vercel build time by Rust compile time (~30–90 s cold, ~5 s incremental with caching).

**Open question:** Does Vercel's build environment support Rust, and if not, is a custom Docker build image feasible?

#### Option B: Gitignore `docs/blog/` and `docs/salt/`, treat `web/content/` as the authoritative source

Authors write directly in `web/content/posts/` (Markdown) and `web/content/booknotes/` (JSON). The `docs/blog/` and `docs/salt/` directories are either removed or kept only as historical archive. The Rust `sync_content` step is removed from the author workflow entirely.

**Pros:**
- Eliminates the sync step and its CRLF drift problem.
- No Rust toolchain dependency at build time for content.
- Simpler mental model: what is committed is what is built.

**Cons:**
- Booknote authoring moves from Markdown (readable, writable in any editor) to JSON (ugly, error-prone to hand-edit). Authoring a new booknote would require manually constructing the JSON shape.
- The `sync_fontbook.rs` script reads from `docs/blog/` and `docs/salt/` to build the CJK font subset. Moving the source of truth would require updating that script as well.
- Removes the Markdown-based editing ergonomics for booknotes that the current `docs/salt/*.md` format provides.

#### Option C: Keep both directories, enforce sync via CI (recommended)

A GitHub Actions step runs `cargo run --bin sync_content` after every push to `master` (or on PRs), then runs `git diff --exit-code web/content/` to verify the synced output matches what is committed. If it does not, the step fails and the author is told to run sync locally before pushing. Authors continue to write in `docs/`; `web/content/` stays committed.

**Pros:**
- No changes to the build environment or Vercel config — Rust is only needed locally and in CI, not at Vercel build time.
- Catches the current CRLF drift and any future divergence before merge.
- Keeps Markdown authoring ergonomics for booknotes (the `docs/salt/*.md` flow).
- Minimal disruption to the existing architecture.

**Cons:**
- Authors must remember to run `sync_content` locally before pushing; the CI failure is the only enforcement.
- The `git diff` check will be permanently noisy for CRLF drift unless a `.gitattributes` rule normalizes line endings (`*.md text eol=lf`), which should be added as a prerequisite. **Note:** adding that rule will cause Git to rewrite all 23 currently-CRLF files in `web/content/posts/` on the next `git checkout` or `git add`, producing a one-time commit that touches all 23 files — this is the correct cleanup but should not be a surprise.
- The onepiece workflow remains ad-hoc: `data/op_sanitized.json` is absent from the repo, so CI cannot verify or regenerate it.

### Recommendation

**Option C — keep both directories, enforce sync via CI** is the lowest-risk path. The current architecture is sound: authors work in Markdown, the sync produces derived JSON, and Astro consumes the derived output. The only failure is the absence of any enforcement. A CI check (`cargo run --bin sync_content && git diff --exit-code web/content/`) plus a `.gitattributes` line-ending rule would close both the enforcement gap and the CRLF noise in one pass. Option A is architecturally cleaner but requires solving the Rust-in-Vercel-build and missing-`data/`-directory problems before it is viable. Option B degrades booknote authoring ergonomics with no offsetting benefit.

**Severity: medium** — The drift does not break the site (CRLF is transparent to Astro, booknotes are in sync, onepiece data is committed), but the workflow has no enforcement and the onepiece source artifact is unrepresentable from the current checkout. Left unaddressed, a new contributor or a clean clone cannot reproduce the `web/content/` state from source, which is a meaningful DX and reproducibility gap.

### A. Astro 5 features the codebase does not use

**A1 — `astro:env` (type-safe environment variables)**

What it is: A first-party Astro 5 API that lets you declare environment variables in `astro.config.mjs` with types, `access: "public" | "secret"`, and optional default values; the schema is validated at build time and the variables are imported from `astro:env/client` or `astro:env/server` rather than accessed via `import.meta.env.*`.

**A2 — Server islands (`server:defer`)**

What it is: An Astro 5 primitive that renders a component in a deferred server request after the main page HTML has been sent, using `<Component server:defer />` — similar to React Suspense but native to Astro and framework-agnostic.

What it is: Astro 5's Content Layer API allows defining custom `loader` objects (not just the built-in `glob` and `file` loaders) that can fetch data from any source — REST APIs, CMS, databases — and expose it as a typed content collection.

Why it might help: The site is a small personal portfolio with fast static pages — a subtle fade or slide between pages would feel more polished without adding any JS weight on non-transitioning loads (the `ClientRouter` script is small and only activates on navigation). ### Phase 4 — BE fix

No direct findings from this audit — Phase 4 scope is the Lightsail Rust backend, which is currently down. Audit findings that have direct Phase 4 relevance:

- The `/api/op/today` endpoint computation is deterministically reproducible client-side from the bundled `characters.json` (seed = `year * 10000 + month * 100 + day` UTC → index into character array). Eliminating the fetch removes the dependency on the backend for the `/op/` route entirely and resolves the dead `loading`/`error` state that currently shows no UI when the backend is down (Axis 3 — OnePiece deep-dive). This is the recommended path; it makes Phase 4 a narrower problem (back-end fix needed only for features other than the daily character pick).
- If the fetch is retained rather than eliminated, `vercel.json`'s rewrite of `/api/(.*)` → `https://api.k27dong.com/api/$1` is the single point of failure — no health-check, no fallback. The `api.k27dong.com` domain lapsing silently breaks `/op/` (Axis 4).
- `data/op_sanitized.json` is absent from the repo; the onepiece character data cannot be regenerated from the current checkout without the raw scrape artifact. If the backend is rebuilt, the scrape pipeline's output needs to be committed or its source tracked (Axis 5).

## Open Questions

- **`@astrojs/mdx` — is any `.mdx` content planned?** The integration is wired in `astro.config.mjs` and loads unconditionally, but zero `.mdx` files exist in `web/`. If no MDX content surfaces are planned, the integration should be removed. If MDX is planned, no action needed — bump to 5.x as part of the Astro 6 migration sprint (Axis 1).
- **`data/op_sanitized.json` — where does it come from and where is its raw input?** The file is absent from the repository; the `data/` directory does not exist in the current checkout. `sync_onepiece()` mentions a multi-step scrape pipeline (`data/op_original.json` → sanitize → `data/op_sanitized.json`), but neither file nor the pipeline tooling is committed. A clean checkout cannot reproduce `web/content/onepiece/characters.json` without the external artifact. Where is the scrape pipeline, and should `data/op_sanitized.json` (or a pointer to it) be committed? (Axis 5)
- **Does Vercel's build environment support Rust?** Option A in Axis 5 (gitignore `web/content/`, regenerate at build) requires `cargo run --bin sync_content` to run in Vercel's build environment. Vercel does not install the Rust toolchain by default. A custom install step or a different CI strategy (pre-generate `web/content/` in a GH Action and cache it) would be needed. This is the primary blocker for Option A becoming viable (Axis 5).
- **Booknote JSON schema completeness — does a full booknote JSON file contain `id`, `notenum`, `rating`, and `tags` at the top level?** The checked sample (`国境以南_太阳以西.json`) showed only `author` and `content` in the visible head. The `web/content.config.ts` schema requires `id`, `notenum`, `rating`, and `tags`. If these fields are absent from complete booknote JSON files, schema validation would fail at build time. A full file needs to be checked (Axis 3 — Schema audit).
- **`is_internship` flag intent for `experiences` collection — should it default to `false` silently?** `is_internship: z.boolean().optional().default(false)` silently treats any entry without the field as non-internship. If a new experience entry omits the flag accidentally, no validation error surfaces. Is the silent default the intended authoring contract, or should `is_internship` be required for all future entries? (Axis 3 — Schema audit)
- **`tiny-pinyin` — is it used by anything outside the FE?** In the FE, `tiny-pinyin` is imported only in `web/components/op/OnePiece.tsx` for Pinyin-based character search. The audit found no other reference to it. If the OnePiece search logic were rewritten or the pinyin search removed, this dep would become unused. Confirming there is no off-FE consumer (e.g., a Rust-side or build-time script) validates safe removal if the search changes (Axis 1).
